# docs/spec.md

## 1. 概要
ローカルフォルダ内の画像を再帰的に探索し、ローカルで起動しているエンドポイントに問い合わせることで説明文（caption/description）を生成して、各画像と同一フォルダに txt ファイルとして保存する。
生成結果の txt は Gradio のWeb UIから閲覧・編集できる。
このアプリを captify と命名する。

---

## 2. 実行環境
- 仮想環境は **venv** を使用する
- OSは Windows / macOS / Linux を想定

---

## 3. エンドポイントとエンドポイントに対する動作
- デフォルト：http://127.0.0.1:1234
  UIから変更可能とする。
- エンドポイントは OpenAI互換、LM Studio API が利用できる。
- 採用APIは `/v1/chat/completions` に固定する。
- models を取得し、利用するモデルを明らかにする。また利用するモデルを指定可能とする。
- 画像は `messages[].content` 内の `image_url` に base64 data URL を設定して渡す。
- ストリーミングは利用する（`stream=true`）。

---

## 4. 入力画像
- 指定フォルダを再帰探索
- 対象拡張子：.png, .jpg, .jpeg, .webp, .bmp
- 探索時のシンボリックリンクの扱いは規定しない（don't care）
- 探索時の隠しファイル・隠しディレクトリの扱いは規定しない（don't care）
- 処理順はフルパス昇順で固定し、再現性を担保する

---

## 5. 出力仕様

### 5.1 出力先
画像と同一フォルダ

### 5.2 ファイル名
image01.jpg → image01.txt

### 5.3 内容
モデル生成テキストをそのまま保存する

---

## 6. バックアップ仕様
既存 txt がある場合は、保存前にバックアップファイルを作成する。命名規則は以下の通り。

### 6.1 バックアップファイル名形式
- 元ファイル `image01.txt` のバックアップは、拡張子 `.txt` を 3 桁連番に置き換える。
- 例：
  - `image01.001`
  - `image01.002`
- バックアップファイル名には元の拡張子 `.txt` を残さない。

### 6.2 採番対象の探索条件
- 探索対象は「同一ディレクトリ」かつ「同一ベース名（`image01`）」のファイルのみとする。
- 採番対象に含めるのは `image01.NNN`（`NNN` は 3 桁数字）に一致するファイルのみとし、`image01.txt` は採番候補に含めない。
- 上記の既存バックアップの最大番号を取得し、次番号は `最大番号 + 1` とする。
- 既存バックアップが 1 件も無い場合は `001` から開始する。

### 6.3 採番上限と衝突時の挙動
- 採番上限は `999` とする。
- 次番号が `1000` 以上になる場合は、UI に ERROR を表示し、保存処理をスキップする。
- 算出した次番号のバックアップファイルが既に存在する場合も、UI に ERROR を表示し、保存処理をスキップする。

### 6.4 適用範囲
- 本仕様は主機能1（7章）と主機能2（8章）の保存処理に共通で適用する。

## 7. 主機能1：一括生成

### UI要素
- モデル設定
  - エンドポイント表示・指定　
  - モデル取得ボタン
  - モデル選択プルダウン
- 画像
  - 入力フォルダ指定欄
  - フォルダ内画像プレビュー
- モデル応答
- プロンプト
  - プロンプト
  - プロンプトプリセット
- パラメータ
  - max_tokens
  - temperature
  - top_p
- 実行ボタン
- テストボタン
- ログ表示

### 挙動・動作
- 入力フォルダ内再帰列挙
- 列挙結果はフルパス昇順で処理する
- 保存前バックアップ（6章のバックアップ仕様を適用）
- 保存テキストは「最初の assistant message の text」を抽出して使用する。
- モデル応答にはストリーミングで得られたテキストを逐次表示し、生成完了時に確定した応答へ更新する。
- 正常応答は以下をすべて満たした場合とする。
  - HTTPステータスコードが2xxである。
  - ストリーミング完了後に抽出した「最初の assistant message の text」が非空である（空白のみは空とみなす）。
  - 抽出テキスト長が `max_tokens` を超過しない。
- 異常時は以下の再試行ルールを適用する。
  - 対象：HTTPエラー（2xx以外）、タイムアウト、推論処理中の接続エラー。
  - 再試行回数：最大2回（初回試行を含めて最大3回実行）。
  - 待機秒：再試行前に毎回2秒待機する。
  - 最終判定：再試行上限に達しても正常応答条件を満たさない場合、当該画像はスキップする。
  - 抽出テキストが空文字の場合は再試行せず即時スキップする。
- スキップ時はログに以下の必須項目を含める。
  - 画像パス `file={image_path}`
  - エラー種別 `error_type={error_type}`（`http_error` / `timeout` / `connection_error` / `empty_text`）
  - HTTP status `status={status_code}`（未取得時は `status=none`）
  - モデル名 `model={model_name}`
  - ログ文言形式：`SKIP: error_type={error_type} status={status_code_or_none} model={model_name} file={image_path}`
- 読み込み不可ファイル・破損画像は当該ファイルをスキップし、ログ表示とUI表示の双方にエラーを出す。
  - エラー種別は `error_type=file_read_error` または `error_type=corrupt_image` を使用する。
  - ログ文言は既存のスキップ形式に従い、`SKIP: error_type={error_type} status=none model={model_name} file={image_path}` とする。
  - UI表示には、対象ファイルが読み込み不可または破損画像である旨を明示する。
- テストボタンは最初の画像1枚目のみを問い合わせ、応答を表示する。txtの保存は行わない。
  - 失敗時は「モデル応答」領域に成功レスポンスの代わりにエラーメッセージを表示する。
  - 表示文言は `TEST FAILED: {error_type} status={status_code_or_none} model={model_name}` 形式とし、必要に応じて詳細（タイムアウト、接続失敗理由）を追記する。
  - 同時に「ログ表示」領域にも同内容を1行追記する。
