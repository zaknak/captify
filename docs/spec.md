# docs/spec.md

## 1. 目的・対象ユーザー・ユースケース（What）

### 1.1 目的
ローカルフォルダ内の画像を再帰的に探索し、ローカルで起動している OpenAI 互換エンドポイント（LM Studio API）に問い合わせて説明文（caption/description）を生成し、各画像と同一フォルダに txt ファイルとして保存する。生成済み txt は Gradio の Web UI から閲覧・編集できる。このアプリを captify と命名する。

### 1.2 対象ユーザー
- ローカル環境で画像説明文を一括生成したいユーザー
- 生成済み説明文を画面上で確認・編集して保存したいユーザー

### 1.3 ユースケース
- **主機能1（一括生成）の価値**：複数画像に対して説明文を効率的に生成・保存できる。
- **主機能2（編集）の価値**：生成済みテキストを画像と対応づけて確認・修正し、保存できる。

## 2. 機能要件（必須）（What）

### 2.1 実行環境
- 仮想環境は **venv** を使用する
- OS は Windows / macOS / Linux を想定

### 2.2 エンドポイント
- デフォルト：`http://127.0.0.1:1234`（UI から変更可能）
- OpenAI 互換 API（LM Studio API）を利用する
- 採用 API は `/v1/chat/completions` に固定する
- モデル一覧取得 API は `/v1/models` に固定する
- `models` を取得し、利用するモデルを明らかにする。また利用するモデルを指定可能とする
- 画像は `messages[].content` 内の `image_url` に base64 data URL を設定して渡す
- ストリーミングを利用する（`stream=true`）

### 2.3 入力画像
- 指定フォルダを再帰探索
- 対象拡張子：`.png`, `.jpg`, `.jpeg`, `.webp`, `.bmp`
- 探索時のシンボリックリンクの扱いは規定しない（don't care）
- 探索時の隠しファイル・隠しディレクトリの扱いは規定しない（don't care）
- 処理順はフルパス昇順で固定し、再現性を担保する

### 2.4 保存結果

#### 2.4.1 出力先
- 画像と同一フォルダ

#### 2.4.2 ファイル名
- `image01.jpg` → `image01.txt`

#### 2.4.3 内容
- モデル生成テキストをそのまま保存する

#### 2.4.4 バックアップ仕様（主機能1/主機能2 共通）
- 既存 txt がある場合は、保存前にバックアップファイルを作成する
- 元ファイル `image01.txt` のバックアップは、拡張子 `.txt` を 3 桁連番に置き換える
  - 例：`image01.001`, `image01.002`
- バックアップファイル名には元の拡張子 `.txt` を残さない
- 探索対象は「同一ディレクトリ」かつ「同一ベース名（`image01`）」のファイルのみとする
- 採番対象に含めるのは `image01.NNN`（`NNN` は 3 桁数字）に一致するファイルのみとし、`image01.txt` は採番候補に含めない
- 既存バックアップの最大番号を取得し、次番号は `最大番号 + 1` とする
- 既存バックアップが 1 件も無い場合は `001` から開始する
- 採番上限は `999` とする

### 2.5 成功条件
- 正常応答は以下をすべて満たした場合とする
  - HTTP ステータスコードが 2xx である
  - ストリーミング完了後に抽出した「最初の assistant message の text」が非空である（空白のみは空とみなす）
  - 抽出テキスト長が `max_tokens` を超過しない

### 2.6 主機能1：一括生成

#### UI要素
- モデル設定
  - エンドポイント表示・指定
  - モデル取得ボタン
  - モデル選択プルダウン
- 画像
  - 入力フォルダ指定欄
  - フォルダ内画像プレビュー
- モデル応答
- プロンプト
  - プロンプト
  - プロンプトプリセット
    - 初期プリセットは以下 3 件を定義する
      1. `事実ベース（簡潔）`
      2. `商品説明（EC向け）`
      3. `SNS向け（自然文）`
    - プリセットの保持方式は、ユーザーが編集可能な JSON ファイルとする
- パラメータ
  - `max_tokens`
  - `temperature`
  - `top_p`
- 実行ボタン
- テストボタン
- ログ表示

#### 入出力
- 入力フォルダ内を再帰列挙し、列挙結果をフルパス昇順で処理する
- プロンプトプリセット選択時に、プリセット本文をプロンプトテキスト欄へ反映（コピー）する
- 保存テキストは「最初の assistant message の text」を抽出して使用する
- モデル応答はストリーミングテキストを逐次表示し、生成完了時に確定応答へ更新する
- テストボタンは最初の画像 1 枚目のみを問い合わせ、応答を表示する。txt の保存は行わない

#### 本文から参照する運用ルール
- 例外・再試行・ログ・スキップ・タイムアウトは **3 章** を参照

### 2.7 主機能2：編集（仕様保持・実装スコープ外）
> **注記**：主機能2は本仕様上の定義を保持するが、現状は「未実装対象」であり実装スコープ外とする。

#### UI要素
- フォルダ指定
- 画像一覧
- 画像プレビュー
- テキスト編集欄
- 保存ボタン

#### 入出力
- 既存ならバックアップ後上書き（`2.4.4 バックアップ仕様` を適用）
- 無ければ新規作成

## 3. 非機能・例外・運用（How）

### 3.1 エラー分類
- 入力フォルダ検証エラー
  - `folder_not_specified`
  - `folder_not_found`
  - `not_a_directory`
  - `permission_denied`
- 推論・通信系エラー
  - `http_error`
  - `timeout`
  - `connection_error`
  - `empty_text`
- ファイル/画像エラー
  - `file_read_error`
  - `corrupt_image`

### 3.2 再試行
- 対象：HTTP エラー（2xx 以外）、タイムアウト、接続エラー
- 再試行回数：最大 2 回（初回試行を含めて最大 3 回実行）
- 待機秒：再試行前に毎回 2 秒待機する
- 最終判定：再試行上限に達しても成功条件を満たさない場合、当該処理は失敗とする
- 抽出テキストが空文字の場合は再試行せず即時スキップする
- モデル一覧取得失敗時の再試行も本節ルールを適用する

### 3.3 ログ形式
- スキップ時は以下の必須項目を含める
  - `file={image_path}`
  - `error_type={error_type}`
  - `status={status_code}`（未取得時は `status=none`）
  - `model={model_name}`
- スキップログ文言形式
  - `SKIP: error_type={error_type} status={status_code_or_none} model={model_name} file={image_path}`
- 入力フォルダ検証ログ文言
  - `ERROR: input_folder_validation error_type=... folder={folder}`
- 対象画像 0 件時ログ文言
  - `WARNING: no_target_images folder={folder} count=0 action=stop_without_save`

### 3.4 スキップ条件
- 入力フォルダ未指定
  - UI 表示（ERROR）：`ERROR: 入力フォルダが未指定です。folder={folder}`
- 入力フォルダ不存在
  - UI 表示（ERROR）：`ERROR: 入力フォルダが存在しません。folder={folder}`
- 入力パスがディレクトリでない
  - UI 表示（ERROR）：`ERROR: 入力パスがディレクトリではありません。folder={folder}`
- 入力フォルダアクセス拒否
  - UI 表示（ERROR）：`ERROR: 入力フォルダにアクセスできません。folder={folder}`
- 入力フォルダ検証成功後、対象画像が 0 件
  - API 呼び出し・推論処理を行わず即時終了
  - UI 表示（WARNING）：`WARNING: 対象画像が見つかりませんでした。folder={folder} count=0`
  - 保存処理（txt 新規作成・上書き・バックアップ）を実施しない
- 読み込み不可ファイル・破損画像
  - 当該ファイルをスキップし、UI 表示とログ表示の双方にエラーを出す
- 再試行上限到達後も成功条件を満たさない場合
  - 当該画像をスキップする

### 3.5 タイムアウト
- タイムアウトは再試行対象のエラーとして扱う
- タイムアウト詳細は失敗時表示文言に含めてよい

### 3.6 モデル取得運用
- 取得先は `{endpoint}/v1/models`
- 正常時は、取得したモデル一覧で「モデル選択プルダウン」を更新し、先頭モデルを初期選択する
- 再試行上限到達で失敗した場合
  - UI 表示（ERROR）：`ERROR: モデル一覧の取得に失敗しました。endpoint={endpoint} retries=2 wait=2s`
  - モデル選択プルダウンは空のまま（選択不可）
  - プレースホルダー表示は `モデル未取得`
- モデル未取得状態（モデル選択プルダウンが空）では「実行」「テスト」は実行不可
  - API 呼び出しを行わず UI に ERROR を表示
  - 文言：`ERROR: モデルが未選択です。先にモデル取得を実行してください。`
  - 手動入力によるモデル名指定は許可しない

### 3.7 テストボタン運用
- 入力フォルダ検証に成功しても対象画像が 0 件の場合
  - API 呼び出しは行わない
  - 「モデル応答」領域に `TEST SKIPPED: 対象画像が0件のためテストを実行しません。folder={folder} count=0` を表示
  - 「ログ表示」領域にも同内容を 1 行追記
- 失敗時
  - 「モデル応答」領域に成功レスポンスの代わりにエラーメッセージを表示
  - 表示文言：`TEST FAILED: {error_type} status={status_code_or_none} model={model_name}`
  - 必要に応じて詳細（タイムアウト、接続失敗理由）を追記
  - 「ログ表示」領域にも同内容を 1 行追記
