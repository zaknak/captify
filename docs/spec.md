# docs/spec.md

## 1. 目的・対象ユーザー・ユースケース（What）

### 1.1 目的
ローカルフォルダ内の画像を再帰的に探索し、ローカルで起動している OpenAI 互換エンドポイント（LM Studio API）に問い合わせて説明文（caption/description）を生成し、各画像と同一フォルダに txt ファイルとして保存する。生成済み txt は Gradio の Web UI から閲覧・編集できる。このアプリを captify と命名する。

### 1.2 対象ユーザー
- ローカル環境で画像説明文を一括生成したいユーザー
- 生成済み説明文を画面上で確認・編集して保存したいユーザー

### 1.3 ユースケース
- **主機能1（一括生成）の価値**：複数画像に対して説明文を効率的に生成・保存できる。
- **主機能2（編集）の価値**：生成済みテキストを画像と対応づけて確認・修正し、保存できる。

## 2. 機能要件（必須）（What）

### 2.1 実行環境
- 仮想環境は **venv** を使用する
- OS は Windows / macOS / Linux を想定

### 2.2 エンドポイント
- デフォルト：`http://127.0.0.1:1234`（UI から変更可能）
- OpenAI 互換 API（LM Studio API）を利用する
- 採用 API は `/v1/chat/completions` に固定する
- モデル一覧取得 API は `/v1/models` に固定する
- `models` を取得し、利用するモデルを明らかにする。また利用するモデルを指定可能とする
- モデル選択は `/v1/models` レスポンス `data[].id` を表示値兼内部値として使用する
- 画像は `messages[].content` 内の `image_url` に base64 data URL を設定して渡す
- ストリーミング利用の有効/無効をUIから切り替え可能とする（デフォルトは有効）
- `/v1/chat/completions` のストリーミングチャンクは `data: {json}` 行のみを処理する
- ストリーミング時は `choices[0].delta.content` の文字列を受信順に連結し、`data: [DONE]` 受信を完了条件とする

### 2.3 入力画像
- 指定フォルダを再帰探索
- 対象拡張子：`.png`, `.jpg`, `.jpeg`, `.webp`, `.bmp`
- 探索時のシンボリックリンクの扱いは規定しない（don't care）
- 探索時の隠しファイル・隠しディレクトリの扱いは規定しない（don't care）
- 処理順はフルパス昇順で固定し、再現性を担保する

#### 2.3.1 画像リサイズと base64 エンコード
- 画像送信前に、総画素数を検査する
- 閾値は以下とする
  - 既定の最大画素数: `1.6384` メガピクセル（= `1638400` px）
- 最大画素数はUIのパラメータ `max_image_megapixels` として指定可能とする
- `max_image_megapixels` の入力単位はメガピクセル（MP）とし、小数を受け付ける
- 画像の総画素数が閾値を超える場合は、アスペクト比を維持した縮小リサイズを行う
- 縮小率は `min(1.0, sqrt(最大画素数px / 元画像画素数))` とする
- リサイズ後の幅・高さはともに `1` px 以上とする
- 閾値内の画像はリサイズせず、そのまま base64 data URL 化する
- 画像フォーマットは入力拡張子に対応する MIME を維持する（`.jpg` と `.jpeg` は `image/jpeg`）
- `.jpg`/`.jpeg` でアルファチャンネルを持つ場合は `RGB` に変換して保存する
- 画像破損・読み込み失敗時は既存仕様どおり SKIP とし、処理全体は継続する
- リサイズを実行した場合、INFOログに `image_resized` を出力する


### 2.4 保存結果

#### 2.4.1 出力先
- 画像と同一フォルダ

#### 2.4.2 ファイル名
- `image01.jpg` → `image01.txt`

#### 2.4.3 内容
- モデル生成テキストをそのまま保存する

#### 2.4.4 バックアップ仕様（主機能1/主機能2 共通）
- 既存 txt がある場合は、保存前にバックアップファイルを作成する
- 元ファイル `image01.txt` のバックアップは、拡張子 `.txt` を 3 桁連番に置き換える
  - 例：`image01.001`, `image01.002`
- バックアップファイル名には元の拡張子 `.txt` を残さない
- 探索対象は「同一ディレクトリ」かつ「同一ベース名（`image01`）」のファイルのみとする
- 採番対象に含めるのは `image01.NNN`（`NNN` は 3 桁数字）に一致するファイルのみとし、`image01.txt` は採番候補に含めない
- 既存バックアップの最大番号を取得し、次番号は `最大番号 + 1` とする
- 既存バックアップが 1 件も無い場合は `001` から開始する
- 採番上限は `999` とする

### 2.5 成功条件
- 正常応答は以下をすべて満たした場合とする
  - HTTP ステータスコードが 2xx である
  - ストリーミング完了後に抽出した「最初の assistant message の text」が非空である（空白のみは空とみなす）
  - `max_tokens` はリクエスト時に指定するが、レスポンスのトークン数は成功/失敗判定に用いない（don't care）

### 2.6 主機能1：一括生成

#### UI要素

  - モデル取得ボタン
  - モデル選択プルダウン
- 画像
  - 入力フォルダ指定欄
  - フォルダ内画像プレビュー
  - フォルダ内画像プレビュー更新ボタン
- モデル応答
- プロンプト
  - プロンプト
  - プロンプトプリセット
    - 初期プリセットは以下 3 件を定義する
      1. `事実ベース（簡潔）`
      2. `商品説明（EC向け）`
      3. `SNS向け（自然文）`
    - プリセットの保持方式は、ユーザーが編集可能な JSON ファイルとする
    - プリセット編集
      - プリセット名入力欄
      - 新規登録ボタン（現在のプロンプト本文を新規プリセットとして保存）
      - 選択中を更新ボタン（現在のプロンプト本文で選択中プリセットを上書き）
      - 選択中を削除ボタン（選択中プリセットを削除）
- パラメータ
  - `max_tokens`
  - `temperature`
  - `top_p`
  - `max_image_megapixels`（既定: `1.6384`）
- 実行ボタン
- テストボタン
- ログ表示

#### 入出力
- 入力フォルダ内を再帰列挙し、列挙結果をフルパス昇順で処理する
- プロンプトプリセット選択時に、プリセット本文をプロンプトテキスト欄へ反映（コピー）する
- 新規登録時、プリセット名が未入力または重複している場合は保存せずERRORを表示する
- 選択中を更新時、選択中プリセットが無い場合は更新せずERRORを表示する
- 選択中を削除時、選択中プリセットが無い場合は削除せずERRORを表示する
- プリセット編集（新規登録/更新/削除）成功時は `presets.json` に即時反映し、ドロップダウン選択肢を更新する
- 削除成功後は、残存プリセットの先頭を選択状態にし、本文をプロンプト欄へ反映する
- 最後の1件は削除不可とし、削除操作時はERRORを表示する
- 保存テキストは「最初の assistant message の text」を抽出して使用する
- ストリーミング有効時、モデル応答は逐次表示し、生成完了時に確定応答へ更新する
- ストリーミング無効時、モデル応答は最終確定応答のみ表示する
- 生成テキストは最終連結後に `strip()` を行い、空文字なら `empty_text` として即時スキップする（再試行しない）
- テストボタンは最初の画像 1 枚目のみを問い合わせ、応答を表示する。txt の保存は行わない
- フォルダ内画像プレビューは入力フォルダ変更時に自動更新せず、更新ボタン押下時に再帰探索して更新する
- プレビュー更新時に対象画像が 0 件の場合は `3.3` / `3.6` 準拠の WARNING を表示し、API 呼び出しは行わない

### 受け入れ条件

#### 正常系（一括生成：再帰探索・生成・保存・バックアップ）
- 入力フォルダ検証成功かつ対象画像が 1 件以上あるとき、対象拡張子の画像を再帰列挙し、フルパス昇順で処理できること
- 各画像で成功条件を満たす応答を得た場合、「最初の assistant message の text」を保存テキストとして画像と同一フォルダに `{画像ベース名}.txt` で保存できること
- 既存 `{画像ベース名}.txt` がある場合、上書き保存前に `2.4.4 バックアップ仕様` に従ってバックアップを作成できること
- モデル応答は、生成中のストリーミング表示と生成完了後の確定応答表示が行われること

#### テストボタン
- 入力フォルダ検証成功かつ対象画像が 1 件以上あるとき、先頭 1 画像に対してのみ推論を実行し、応答を表示できること
- テストボタン実行時は、txt の新規作成・上書き・バックアップを一切行わないこと

#### 異常系（判定可能な UI 表示）
- 入力フォルダ未指定時、UI に `ERROR: 入力フォルダが未指定です。folder={folder}` を表示すること
- 入力フォルダ不存在時、UI に `ERROR: 入力フォルダが存在しません。folder={folder}` を表示すること
- 入力パスがディレクトリでない時、UI に `ERROR: 入力パスがディレクトリではありません。folder={folder}` を表示すること
- 入力フォルダ検証成功後に対象画像が 0 件の時、UI に `WARNING: 対象画像が見つかりませんでした。folder={folder} count=0` を表示し、保存処理を実施しないこと
- モデル未取得状態で「実行」「テスト」を押した時、API 呼び出しを行わず UI に `ERROR: モデルが未選択です。先にモデル取得を実行してください。` を表示すること
- モデル一覧取得が再試行上限到達で失敗した時、UI に `ERROR: モデル一覧の取得に失敗しました。endpoint={endpoint} retries=2 wait=2s` を表示すること
- テストボタンで推論失敗時、「モデル応答」および「ログ表示」に `TEST FAILED: {error_type} status={status_code_or_none} model={model_name}` を表示すること
- プリセット新規登録時、成功したらドロップダウン選択肢に追加され、登録したプリセットが選択状態になること
- プリセット更新時、成功したら選択中プリセット本文が現在のプロンプト本文で置換されること
- プリセット削除時、成功したら当該プリセットが選択肢から除去され、残存先頭プリセットが選択されること
- 最後の1件削除を試行した時、削除は行われずERRORが表示されること

#### 本文から参照する運用ルール
- 例外・再試行・ログ・スキップ・タイムアウトは **3 章** を参照

## 3. エラー処理共通ポリシー

### 3.1 エラー分類
- 入力フォルダ検証エラー
  - `folder_not_specified`
  - `folder_not_found`
  - `not_a_directory`
  - `permission_denied`
- 推論・通信系エラー
  - `http_error`
  - `timeout`
  - `connection_error`
  - `empty_text`
- ファイル/画像エラー
  - `file_read_error`
  - `corrupt_image`
  - `backup_limit_exceeded`

### 3.2 UI表示レベル
- `ERROR`
  - 処理継続不可、または対象処理を失敗として扱うエラー
  - 例：入力フォルダ検証失敗、モデル未取得、再試行上限到達、バックアップ採番上限到達
- `WARNING`
  - 全体停止は不要だが、対象スキップまたは保存未実施を伴う注意
  - 例：対象画像 0 件、読み込み不可ファイル、破損画像
- `INFO`
  - 状態通知・進捗通知（失敗ではない）

### 3.3 ログ必須項目・フォーマット
- エラー・スキップ・失敗時のログは以下の必須項目を含める
  - `error_type={error_type}`
  - `status={status_code}`（未取得時は `status=none`）
  - `model={model_name}`（未確定時は `model=none`）
  - `file={image_path}`（対象ファイルが無い処理では `file=none`）
- スキップログ文言形式
  - `SKIP: error_type={error_type} status={status_code_or_none} model={model_name_or_none} file={image_path_or_none}`
- 入力フォルダ検証ログ文言
  - `ERROR: input_folder_validation error_type=... folder={folder}`
- 対象画像 0 件時ログ文言
  - `WARNING: no_target_images folder={folder} count=0 action=stop_without_save`

### 3.4 再試行の基本形式
- 対象：HTTP エラー（2xx 以外）、タイムアウト、接続エラー
- 再試行回数：最大 2 回（初回試行を含めて最大 3 回実行）
- 待機秒：再試行前に毎回 2 秒待機する
- 最終判定：再試行上限に達しても成功条件を満たさない場合、当該処理は失敗とする
- 抽出テキストが空文字の場合は再試行せず即時スキップする
- モデル一覧取得失敗時の再試行も本節ルールを適用する

### 3.5 ファイルI/O系の共通ルール
- 読み込み不可ファイル（`file_read_error`）
  - 当該ファイルをスキップし、UI 表示とログ表示の双方にエラーを出す
- 破損画像（`corrupt_image`）
  - 当該ファイルをスキップし、UI 表示とログ表示の双方にエラーを出す
- バックアップ採番上限（`backup_limit_exceeded`）
  - `2.4.4 バックアップ仕様` の上限 `999` 到達時は保存処理を失敗とする
  - UI 表示（ERROR）とログ表示の双方に `error_type=backup_limit_exceeded` を含める

### 3.6 スキップ条件
- 入力フォルダ未指定
  - UI 表示（ERROR）：`ERROR: 入力フォルダが未指定です。folder={folder}`
- 入力フォルダ不存在
  - UI 表示（ERROR）：`ERROR: 入力フォルダが存在しません。folder={folder}`
- 入力パスがディレクトリでない
  - UI 表示（ERROR）：`ERROR: 入力パスがディレクトリではありません。folder={folder}`
- 入力フォルダアクセス拒否
  - UI 表示（ERROR）：`ERROR: 入力フォルダにアクセスできません。folder={folder}`
- 入力フォルダ検証成功後、対象画像が 0 件
  - API 呼び出し・推論処理を行わず即時終了
  - UI 表示（WARNING）：`WARNING: 対象画像が見つかりませんでした。folder={folder} count=0`
  - 保存処理（txt 新規作成・上書き・バックアップ）を実施しない
- 再試行上限到達後も成功条件を満たさない場合
  - 当該画像をスキップする

### 3.7 タイムアウト
- タイムアウトは再試行対象のエラーとして扱う
- タイムアウト詳細は失敗時表示文言に含めてよい

## 4. 非機能・運用（How）
- ログは INFO / WARNING / ERROR を使い分ける。
- 主機能1および将来の主機能2のエラー処理は、重複定義を避けるため **3 章 エラー処理共通ポリシー** を優先適用する。
- Gradio の `allowed_paths` は `allowed_paths.json` と起動オプション `--allowed-path`（複数指定可）で指定可能とする。
- `allowed_paths.json` の形式は `{ "allowed_paths": ["/path/a", "/path/b"] }` とし、適用順は「JSON → 起動オプション」、重複は除去する。
- `allowed_paths.json` が存在しない場合は空配列として扱い起動を継続し、形式不正時は `ValueError` として起動失敗とする。

## 7. 主機能1固有の詳細運用（3章準拠）

### 7.1 モデル取得運用
- 取得先は `{endpoint}/v1/models`
- 正常時は、`2.2` で定義した `data[].id` を用いて「モデル選択プルダウン」を更新し、先頭モデルを初期選択する
- 再試行上限到達で失敗した場合
  - UI 表示（ERROR）：`ERROR: モデル一覧の取得に失敗しました。endpoint={endpoint} retries=2 wait=2s`
  - モデル選択プルダウンは空のまま（選択不可）
  - プレースホルダー表示は `モデル未取得`
- モデル未取得状態（モデル選択プルダウンが空）では「実行」「テスト」は実行不可
  - API 呼び出しを行わず UI に ERROR を表示
  - 文言：`ERROR: モデルが未選択です。先にモデル取得を実行してください。`
  - 手動入力によるモデル名指定は許可しない
  - 本節の失敗判定・再試行・ログは **3 章 エラー処理共通ポリシー** に準拠する

### 7.2 テストボタン運用
- 入力フォルダ検証に成功しても対象画像が 0 件の場合
  - API 呼び出しは行わない
  - 「モデル応答」領域に `TEST SKIPPED: 対象画像が0件のためテストを実行しません。folder={folder} count=0` を表示
  - 「ログ表示」領域にも同内容を 1 行追記
- 失敗時
  - 「モデル応答」領域に成功レスポンスの代わりにエラーメッセージを表示
  - 表示文言：`TEST FAILED: {error_type} status={status_code_or_none} model={model_name}`
  - 必要に応じて詳細（タイムアウト、接続失敗理由）を追記
  - 「ログ表示」領域にも同内容を 1 行追記
  - 本節の失敗判定・再試行・ログは **3 章 エラー処理共通ポリシー** に準拠する
